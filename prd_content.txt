§1  
Project Overview
The Codebase Complexity Analyzer with PDF Reporting is a self-contained Python command-line application designed to perform static analysis on Python codebases. The tool scans a target directory, parses every Python file using Python’s built-in ast (Abstract Syntax Tree) module, computes a comprehensive set of software quality metrics, identifies potential maintainability risks through a warning detection engine, computes per-file and aggregate complexity scores, and produces structured output both in the terminal and as a professionally formatted multi-page PDF report powered by the fpdf2 library.
This project is submitted as the CS50P Final Project and is intentionally scoped to exceed the complexity of any individual problem set. It spans multiple interconnected modules, implements a complete data pipeline from raw source files to polished reports, and includes a thorough pytest-based test suite that validates core algorithmic functions.
Scope Boundaries
In-Scope: Python file discovery, AST-based parsing, LOC counting, function/class enumeration, nesting depth analysis, per-file complexity scoring, warning detection, terminal report, PDF report, summary aggregation, optional JSON export, configurable thresholds, verbose mode.
Out-of-Scope: Refactoring suggestions, cross-file dependency analysis, runtime profiling, support for non-Python languages, GUI front-end, cloud deployment.
§2  
Problem Statement
Python is a dynamically typed language that places high cognitive burden on developers attempting to reason about large codebases. Without tooling, developers and reviewers have no fast, automated way to gauge whether a codebase is growing unwieldy — whether functions are becoming too long, nesting is becoming too deep, or individual files are ballooning beyond maintainable size.
Existing tools such as radon, pylint, and flake8 provide complexity analysis but require additional configuration, produce verbose output that is difficult to share as a document, and have steep learning curves. There is no single lightweight tool that (1) works on a folder of Python files with zero configuration, (2) produces a readable terminal summary, and (3) simultaneously generates a shareable PDF report from one command.
The Codebase Complexity Analyzer fills this gap: a zero-dependency analysis pipeline (beyond fpdf2) that a developer can run on any Python project directory to receive immediate structured feedback.

Functional Requirements
FR-1  Directory Scanner
The tool MUST accept a single positional CLI argument specifying the target folder path. It MUST recursively traverse subdirectories to discover all files with the .py extension. It MUST exclude files that cannot be read due to permission errors, logging a warning instead of aborting. It MUST return an ordered list of absolute paths. If the specified folder does not exist or is empty of .py files, the tool MUST exit with a clear error message and a non-zero exit code.
FR-2  Python File Parser
For each discovered .py file, the tool MUST attempt to parse the file’s source code using Python’s ast.parse() function. Parsing MUST be wrapped in a try/except block that catches SyntaxError and UnicodeDecodeError exceptions. On a parse failure, the file MUST be marked with a parse_error flag and included in the report with a warning note; the tool MUST NOT crash. The resulting ast.Module tree MUST be passed to the metric calculation layer.
FR-3  Code Metrics Calculation
For each successfully parsed file, the following metrics MUST be computed:

FR-5  Complexity Scoring Engine
The scoring engine MUST produce a per-file score in the range [0, 100] where 0 = perfectly clean and 100 = maximally complex/problematic. The score MUST be computed using the following additive penalty formula:
score = 0
# Penalty: long maximum function
if max_func_len > 50:
    score += min(30, (max_func_len - 50) * 0.5)
# Penalty: deep nesting
if nesting_depth > 3:
    score += min(25, (nesting_depth - 3) * 8)
# Penalty: large file
if loc > 300:
    score += min(20, (loc - 300) * 0.05)
# Penalty: excessive functions
if num_functions > 20:
    score += min(15, (num_functions - 20) * 1.0)
# Penalty: high average function length
if avg_func_len > 30:
    score += min(10, (avg_func_len - 30) * 0.4)
score = min(100, round(score, 2))  # clamp to [0, 100]
The OVERALL complexity score MUST be the unweighted arithmetic mean of all per-file scores. Files with parse errors contribute a fixed penalty score of 75 to the overall average.
FR-6  Terminal Report Generator
After completing analysis, the tool MUST print a structured report to stdout. The report MUST include:
A header banner displaying the tool name, target folder, date/time, and total file count.
For each file: relative path, all metric values (M-01 through M-09), a list of triggered warnings with severity, and the per-file complexity score rendered as a colored ASCII bar (green ≤ 30, yellow 31–60, red > 60).
A summary section showing total files analysed, total parse errors, total warnings by severity, average LOC, average function length, and overall complexity score.
FR-7  PDF Report Generator
The tool MUST generate a PDF report using the fpdf2 library. The PDF MUST be written to a configurable output path (default: complexity_report.pdf). The PDF structure MUST conform to the layout defined in §12 of this document. The PDF MUST support automatic pagination and MUST not clip or overflow any text content. All section headers, warnings, and metric tables MUST be formatted with visual hierarchy using fonts, sizes, and line separators.
FR-8  Summary Generator
The summary aggregator MUST compute across all analysed files:
Total files scanned
Total parse errors
Total functions discovered
Total classes discovered
Average LOC per file
Average function length (across all functions in all files)
Overall complexity score (mean of per-file scores)
Distribution of warnings by severity (count of LOW / MEDIUM / HIGH)

Data Flow
The following describes the complete data transformation pipeline from user invocation to report output:
User invokes:  python project.py ./my_project --output report.pdf --verbose
 1. argparse parses CLI args → Namespace(folder='./my_project', output='report.pdf', verbose=True, ...)
 2. get_python_files('./my_project')
    └─ os.walk() yields dirs/files
    └─ Filter *.py extensions
    └─ Returns: ['/abs/path/a.py', '/abs/path/b.py', ...]
 3. For each filepath:
    a) count_lines_of_code(filepath) → int (LOC)
    b) open(filepath).read() → source_code string
    c) ast.parse(source_code) → ast.Module tree  [or None on SyntaxError]
    d) analyze_ast(tree) → {num_functions, num_classes, func_lengths,
                              avg_func_len, max_func_len, nesting_depth, num_loops}
    e) detect_warnings(metrics) → [{id, message, severity}, ...]
    f) calculate_complexity_score(metrics) → float in [0, 100]
    g) Merge all into file_result dict
 4. summarize_results(all_file_results) → summary dict
 5. generate_terminal_report(all_file_results, summary) → stdout
 6. generate_pdf_report(all_file_results, summary, output_path) → .pdf file
 7. If --json flag: json.dump(all_file_results) → results.json

Function Specifications
This section defines the complete contract for every top-level function in project.py. All functions must be implemented exactly as specified to satisfy CS50P compliance and test coverage requirements.
main()
def main() -> None
Purpose
Primary CLI entry point. Orchestrates the full pipeline: parses CLI arguments, invokes the discovery layer, drives the per-file analysis loop, aggregates results, and dispatches terminal and PDF report generation. Exits with code 0 on success, code 1 on fatal errors (missing folder, empty scan result).
Inputs
  •  No direct function arguments. Reads sys.argv via argparse.
  •  CLI args: folder (positional str), --output/-o (str, default: 'complexity_report.pdf'), --verbose/-v (bool flag), --json/-j (bool flag), --max-func-len (int, default: 50), --max-depth (int, default: 3), --no-recurse (bool flag).
Outputs
  •  Returns None. Side effects: writes to stdout, writes PDF file, optionally writes JSON file.
Error Handling
  •  If folder arg is missing: argparse prints usage and exits (code 2).
  •  If folder does not exist: print 'Error: folder not found' and sys.exit(1).
  •  If no .py files found: print 'Error: no Python files found in <folder>' and sys.exit(1).
  •  Any unhandled exception in the analysis loop: caught, logged, and the file is marked with a parse_error flag.
get_python_files(folder)
def get_python_files(folder: str, recursive: bool = True) -> list[str]
Purpose
Recursively (or non-recursively) walks the given folder and returns a sorted list of absolute paths to all files ending in .py. This function is pure relative to the file system: it does not parse or open files.
Inputs
  •  folder (str): Absolute or relative path to the target directory.
  •  recursive (bool): If True, os.walk() is used for full recursion. If False, os.listdir() is used for top-level-only discovery. Default: True.
Outputs
  •  list[str]: Sorted list of absolute .py file paths. Empty list if folder contains no .py files.
Error Handling
  •  If folder does not exist: raises FileNotFoundError (caller must handle).
  •  If a subdirectory raises PermissionError during os.walk(): that subtree is skipped; warning is printed.
  •  Returns [] for an existing but empty folder — does not raise.
Notes
  •  Uses os.path.abspath() to normalise all returned paths.
  •  Sorting is lexicographic to ensure deterministic output across OS platforms.
count_lines_of_code(filepath)
def count_lines_of_code(filepath: str) -> int
Purpose
Opens the file at filepath and counts the number of non-blank, non-comment lines. A blank line is defined as a line containing only whitespace. A comment line is defined as a line whose first non-whitespace character is #. Inline comments (code followed by #) count as code lines.
Inputs
  •  filepath (str): Absolute path to a Python source file.
Outputs
  •  int: Count of qualifying (non-blank, non-comment) lines. Returns 0 for a completely empty or comment-only file.
Error Handling
  •  FileNotFoundError: raised if path does not exist; caller handles.
  •  UnicodeDecodeError: catches and falls back to latin-1 encoding on second read attempt.
  •  PermissionError: propagates to caller for top-level handling.
Notes
  •  Opens file with encoding='utf-8', errors='replace' for robustness.
  •  This function is intentionally kept pure (no AST dependency) to separate metric concerns.
analyze_ast(tree)
def analyze_ast(tree: ast.Module) -> dict
Purpose
Performs a complete walk of the AST tree using ast.walk() and ast.NodeVisitor patterns to extract structural metrics. Returns a dictionary of all computed AST-based metrics.
Inputs
  •  tree (ast.Module): A parsed AST tree returned by ast.parse(). Must not be None.
Outputs
  •  dict with keys: num_functions (int), num_classes (int), func_lengths (list[int]), avg_func_len (float), max_func_len (int), nesting_depth (int), num_loops (int).
Error Handling
  •  If tree is None or not an ast.AST instance: returns a zero-value dict with all metrics set to 0 or [].
  •  If a node lacks lineno/end_lineno (rare in malformed AST): that function is counted but its length recorded as 0.
Notes
  •  Nesting depth is computed via a recursive descent that tracks depth counter through For/While/AsyncFor nodes.
  •  AsyncFunctionDef nodes are counted alongside FunctionDef for num_functions.
detect_warnings(metrics)
def detect_warnings(metrics: dict, thresholds: dict | None = None) -> list[dict]
Purpose
Evaluates the provided metrics dictionary against the warning thresholds (default or overridden) and returns a list of warning objects. Each warning is a dict with keys: id, message, severity.
Inputs
  •  metrics (dict): Output from analyze_ast() merged with count_lines_of_code() and file_size.
  •  thresholds (dict | None): Optional override dict. If None, uses the module-level DEFAULT_THRESHOLDS constant.
Outputs
  •  list[dict]: Each element has keys id (str, e.g. 'W-01'), message (str), severity (str: 'LOW'/'MEDIUM'/'HIGH'). Returns [] if no warnings are triggered.
Error Handling
  •  If metrics dict is missing an expected key: that warning check is skipped and a LOW-severity warning 'Missing metric: <key>' is appended.
  •  If thresholds contains non-numeric values: raises ValueError with descriptive message.
calculate_complexity_score(metrics)
def calculate_complexity_score(metrics: dict) -> float
Purpose
Applies the additive penalty formula defined in FR-5 to produce a per-file complexity score in the range [0.0, 100.0]. This is a pure function with no I/O side effects.
Inputs
  •  metrics (dict): Merged file metrics dict containing: max_func_len, nesting_depth, loc, num_functions, avg_func_len.
Outputs
  •  float: Score in [0.0, 100.0], rounded to 2 decimal places.
Error Handling
  •  If any metric key is missing: that penalty term is skipped (treated as 0).
  •  If all metrics are 0 or below threshold: returns 0.0 (no penalties apply).
  •  Score is always clamped to max 100.0 regardless of extreme input values.
Notes
  •  This function must be deterministic: same input always produces same output.
  •  Penalty weights are read from the SCORING_CONFIG module-level dict, enabling future recalibration.
generate_terminal_report(results, summary)
def generate_terminal_report(results: list[dict], summary: dict) -> None
Purpose
Formats and prints a complete analysis report to stdout. Uses ANSI escape codes for coloured severity indicators and score bars. Outputs per-file sections followed by an aggregate summary.
Inputs
  •  results (list[dict]): List of per-file result dicts, each containing: filepath, loc, num_functions, num_classes, func_lengths, avg_func_len, max_func_len, nesting_depth, num_loops, file_size, warnings, score.
  •  summary (dict): Aggregate statistics produced by summarize_results().
Outputs
  •  Returns None. Side effects: writes formatted text to sys.stdout.
Error Handling
  •  If results is empty: prints 'No files analysed.' and returns immediately.
  •  If a result dict is missing a key: prints '[N/A]' for that metric rather than raising.
  •  ANSI codes are disabled automatically when stdout is not a TTY (e.g. pipe to file).
generate_pdf_report(results, summary, output_path)
def generate_pdf_report(results: list[dict], summary: dict, output_path: str) -> None
Purpose
Constructs a multi-page PDF document using fpdf2 and writes it to output_path. The PDF structure conforms exactly to the layout defined in §12 of this PRD: Title Page, Analysis Section (per-file), Warnings Section, and Summary Section.
Inputs
  •  results (list[dict]): Same structure as generate_terminal_report() input.
  •  summary (dict): Aggregate statistics dict.
  •  output_path (str): Target file path for the PDF. Must end in .pdf.
Outputs
  •  Returns None. Side effects: writes binary PDF data to output_path.
Error Handling
  •  If output_path directory does not exist: raises FileNotFoundError before creating any FPDF object.
  •  If output_path does not end in .pdf: raises ValueError('Output path must end in .pdf').
  •  If fpdf2 raises an internal error during cell rendering: wraps in RuntimeError with original message.
  •  On success, prints 'PDF saved to <output_path>' to stdout.
Notes
  •  Uses FPDF.add_page() with automatic pagination; never hard-codes page breaks.
  •  Sets PDF metadata: title, author, creator fields using FPDF object attributes.
summarize_results(results)
def summarize_results(results: list[dict]) -> dict
Purpose
Aggregates a list of per-file result dicts into a single summary dictionary. Pure function with no I/O side effects.
Inputs
  •  results (list[dict]): Complete list of per-file result dicts. May be empty.
Outputs
  •  dict with keys: total_files (int), parse_errors (int), total_functions (int), total_classes (int), avg_loc (float), avg_func_len (float), overall_score (float), warnings_by_severity (dict: LOW/MEDIUM/HIGH → int).

§16  
Security Considerations
As a local static analysis CLI tool, the attack surface is minimal. The following considerations apply:
The tool reads source files from disk but NEVER executes them. ast.parse() performs syntactic parsing only — no eval(), exec(), or import of user code is performed at any point.
The tool does not make network requests. No data is transmitted externally.
File path inputs are sanitised via os.path.abspath() and os.path.realpath() to prevent path traversal exploits in the folder argument.
PDF output is written to the path specified by the user; the tool validates the extension (.pdf) but does not restrict the directory. Users should be aware of where the output is written.
The tool does not store or log source code content to disk (beyond the PDF report which contains only metrics and warnings, not source code).
